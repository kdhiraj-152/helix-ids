# Hash Authority Specification

## Scope

This document defines the canonical specification for all hash types used in
the HELIX-IDS governance, manifest, and result payloads. It establishes which
component is the authoritative source for each hash, how each hash is
generated, validated, and how conflicts are resolved.

All hashes in this system use SHA-256 with canonical JSON encoding
(`sort_keys=True`, `separators=(",",":")`, UTF-8) unless otherwise specified.

---

## 1. Hash Types

### 1.1 Raw Hash (`raw_hash`)

| Attribute            | Value                                                      |
|----------------------|------------------------------------------------------------|
| Authoritative source | `helix_ids.governance.fingerprinting.build_dataset_manifest_hash` |
| Generated by         | Dataset preparation pipeline                               |
| Computed from        | Sorted file metadata + SHA-256 per-file content hash       |
| Payload structure    | `{"files": [{"path": str, "size": int, "sha256": str}]}`   |
| Appears in           | Manifest top-level `raw_hash`; Result `lineage.dataset_hashes.raw` |

**Generation requirements**:
- Files must be sorted lexicographically by path string before hashing.
- Each file entry includes its absolute/relative path string, byte size, and
  per-file SHA-256 digest.
- The outer hash is the SHA-256 of the canonical JSON encoding of the file
  manifest.

**Validation requirements**:
- Re-derive the hash from the raw dataset files.
- Compare to the stored value; any mismatch is a contract violation.

---

### 1.2 Processed Hash (`processed_hash`)

| Attribute            | Value                                                      |
|----------------------|------------------------------------------------------------|
| Authoritative source | `helix_ids.data.feature_harmonization` (processed artifact writer) |
| Generated by         | Feature harmonization / data processing pipeline           |
| Computed from        | SHA-256 of processed dataset artifacts (Parquet/CSV files) |
| Appears in           | Manifest top-level `processed_hash`; Result `lineage.dataset_hashes.processed` |

**Generation requirements**:
- Computed over the processed (harmonized, feature-engineered) dataset files.
- Must use the same canonical file-manifest method as raw_hash.
- Must be regenerated whenever any processing step changes.

**Validation requirements**:
- Re-derive from processed artifact files.
- Mismatch with stored value indicates processing pipeline drift.

---

### 1.3 Split Hash (`split_hash`)

| Attribute            | Value                                                      |
|----------------------|------------------------------------------------------------|
| Authoritative source | `helix_ids.data.multi_dataset_loader` (split writer)       |
| Generated by         | Data splitting pipeline                                    |
| Computed from        | SHA-256 of split artifact files (train/val/test)           |
| Appears in           | Manifest top-level `split_hash`; Result `lineage.dataset_hashes.split` |

**Generation requirements**:
- Computed over the split artifact files in `data/splits/`.
- Must use the canonical file-manifest method.
- Must be regenerated when split strategy or seed changes.

**Validation requirements**:
- Re-derive from split files.
- Mismatch indicates split reproducibility failure.

---

### 1.4 Primary Hash (`dataset_hash_primary`)

| Attribute            | Value                                                      |
|----------------------|------------------------------------------------------------|
| Authoritative source | Manifest producer (benchmark execution)                    |
| Generated by         | Alias selection from other hashes                          |
| Computed from        | Currently equals `processed_hash`                          |
| Appears in           | Manifest top-level `dataset_hash_primary`; Result `lineage.dataset_hash_primary` and `lineage.dataset_hashes.primary` |

**Generation requirements**:
- Must be set to the hash of the most authoritative dataset stage.
- Currently: `processed_hash` is the primary identity.
- If the authoritative stage changes, this must go through the breaking-change
  approval process.

**Validation requirements**:
- Must match the designated authoritative stage hash.
- Must not be null or empty.

---

### 1.5 Config Hashes (`config_hashes`)

| Attribute            | Value                                                      |
|----------------------|------------------------------------------------------------|
| Authoritative source | Benchmark execution pipeline                               |
| Generated by         | `helix_ids.governance.fingerprinting.canonical_json_hash`  |
| Computed from        | Canonical JSON of each config sub-document                 |
| Appears in           | Manifest top-level `config_hashes`; Result `lineage.config_hashes` |

**Required sub-keys**:

| Sub-key                  | Source                             |
|--------------------------|------------------------------------|
| `helix_config`           | Main HELIX configuration           |
| `training_config`        | Training hyperparameters           |
| `platform_configs`       | Platform deployment configuration  |
| `attack_params`          | Attack simulation parameters       |
| `schema_contract_hash`   | Runtime schema contract hash       |
| `mapping_version`        | Label-mapping version string       |
| `mapping_hash`           | Label-mapping definition hash      |
| `governance_policy_hash` | Governance policy document hash    |
| `run_config_hash`        | Full run configuration hash        |

**Generation requirements**:
- Each sub-key is the SHA-256 of the canonical JSON encoding of the
  corresponding config sub-document.
- All sub-keys must be present for a conforming manifest.

**Validation requirements**:
- All required sub-keys must exist.
- Each hash must be a valid 64-character lowercase hex string.
- Missing or malformed sub-keys are a contract violation.

---

### 1.6 Schema Hash (`schema_hash`)

| Attribute            | Value                                                      |
|----------------------|------------------------------------------------------------|
| Authoritative source | `helix_ids.contracts.schema_contract.compute_schema_hash`  |
| Generated by         | Schema contract module at export time                      |
| Computed from        | `{"schema_version", "feature_order", "input_dim", "binary_output_dim", "family_output_dim"}` |
| Appears in           | Manifest top-level `schema_hash`; Result `lineage.schema_hash`; Config hashes `schema_contract_hash` |

**Generation requirements**:
- Computed from the canonical schema payload in `schema_contract.py`.
- Uses `json.dumps(payload, sort_keys=True, separators=(",",":"))` encoding.
- Any change to feature order, dimensions, or class count changes this hash.

**Validation requirements**:
- Must match `helix_ids.contracts.schema_contract.SCHEMA_HASH`.
- Must also match `config_hashes.schema_contract_hash` when present.
- Mismatch between any two appearances of the schema hash is a contract
  violation.

---

### 1.7 Mapping Hash (`mapping_hash`)

| Attribute            | Value                                                      |
|----------------------|------------------------------------------------------------|
| Authoritative source | `helix_ids.data.feature_harmonization` / label mapping module |
| Generated by         | Label mapping definition module                            |
| Computed from        | Canonical JSON of label-mapping definition                 |
| Appears in           | Manifest top-level `mapping_hash`; Result `lineage.mapping_hash`; Config hashes `mapping_hash` |

**Generation requirements**:
- Computed from the full label-mapping definition (vocabulary, class indices,
  normalization rules).
- Must be regenerated when mapping version changes.

**Validation requirements**:
- All appearances of `mapping_hash` across manifest and result must be
  identical.
- Must correspond to a known `mapping_version`.
- Mismatch between manifest-level and config-hashes-level mapping_hash is a
  contract violation.

## 2. Cross-Hash Consistency Rules

### Machine-enforced (validated by `validate_benchmark_outputs.py`)

1. **Manifest internal**: Within a single manifest, `dataset_hash_primary`
   must equal `processed_hash`.  Enforced in `_check_cross_hash_consistency`.
3. **Config-hash symmetry**: `schema_hash` (top-level) must equal
   `config_hashes.schema_contract_hash`.  Enforced in `_check_cross_hash_consistency`.
4. **Mapping-hash symmetry**: `mapping_hash` (top-level) must equal
   `config_hashes.mapping_hash`.  Enforced in `_check_cross_hash_consistency`.

### Guidance policy (not CI-enforced — require manifest-to-result cross-reference)

2. **Manifest-to-result**: Every hash in the manifest must appear identically
   in the corresponding result's `lineage` object.  This rule requires
   processing result files alongside manifests; `validate_benchmark_outputs.py`
   operates on manifests alone in CI.  It is **enforced by lifecycle verification**
   at the `run_registry.py` level, which performs the manifest-to-result
   cross-reference.  Marked as guidance here to avoid dual-enforcement confusion.
5. **Dataset-hash nesting**: `lineage.dataset_hashes.raw` must equal manifest
   `raw_hash`; same for processed, split, and primary.  Same enforcement
   constraint as Rule 2 — handled by lifecycle verification.

> **Phase 5R Amendment**: Rules 2 and 5 were previously listed as policy without
> machine enforcement.  They are now correctly categorized as guidance policy
> with enforcement delegated to the lifecycle verification layer.  Rules 1, 3, 4
> are fully CI-enforced by `_check_cross_hash_consistency`.

## 3. Conflict Resolution Rules

When a hash mismatch is detected:

1. **Runtime contract hash vs. schema_hash**: The runtime contract module
   (`schema_contract.py`) is the single source of truth. Any other appearance
   must match it.
2. **Manifest vs. result**: The manifest is the authoritative producer; the
   result payload is a copy. If they disagree, the result is invalid.
3. **Computed vs. stored**: A freshly computed hash from source data takes
   precedence over any stored value. If they disagree, the stored value is
   stale and the artifact is invalid.
4. **Ambiguity**: If two stored copies of the same hash disagree and neither
   can be re-derived (e.g., data files absent), the artifact is **rejected**.
   No heuristic reconciliation is permitted.

---

## 4. Hash Format Requirements

- All hashes are SHA-256 digests.
- Encoded as 64-character lowercase hexadecimal strings.
- No whitespace, prefixes (`sha256:`), or truncation.
- Empty strings, `null`, or missing hash fields are always a violation.
