HELIX-IDS

A neural intrusion detection system with a post-hoc control layer that lets you override model predictions at runtime — without retraining.

Status Internal Datasets NSL-KDD · UNSW-NB15 · CICIDS Key file inference_runtime.py

What is HELIX-IDS?

At its core, HELIX-IDS is two things glued together: a neural network that classifies network traffic, and a rule-based layer that can second-guess that classification at inference time.

The neural network part — HelixIDSFull — takes a batch of network flow features and predicts what kind of traffic it is: normal, a DoS attempt, a probe, an R2L attack, and so on. That's the classification job.

The HELIX layer sits on top of that. After the model makes a prediction, HELIX looks at how confident that prediction was (measured by the "margin" between the top two scores). If the model picked a certain class but didn't pick it by a convincing margin, HELIX can swap the answer to the next-best class instead. No retraining, no weight changes — just a runtime decision rule.

Why would you want that? Because retraining a model every time operational requirements change is slow and expensive. Post-hoc control lets you tune the system's behaviour at deploy time. The specific use case here is limiting risky false positives for a target attack family (class 4 by default).

💡
One-liner: HELIX = trained classifier + margin-based override logic that runs at inference without touching model weights.

Data & Features

Datasets

Three benchmark IDS datasets are supported, all configured in src/helix_ids/data/dataset_config.py:

The production model doesn't use all the raw features from any one dataset. Instead, a harmonisation step maps each dataset to a shared 17-dimensional invariant feature vector, defined in feature_mappings.json and referenced from helix_full_config.py.

⚠️
Watch out: helix_full_config.py has input_dim = 18 while helix_ids_full.py defaults to input_dim = 17. These disagree by one. It doesn't cause a crash but it's a latent misconfiguration risk for anything that touches related numeric parameters.

Preprocessing

All preprocessing lives in src/helix_ids/data/preprocessing.py under DataPreprocessor. The steps in order:

  1. Impute missing values using column statistics
  2. Encode categorical features
  3. Scale features (Standard, MinMax, or Robust — configurable)
  4. Clip outliers to a configured range

The fitted scaler stats are stored and reapplied identically at inference. You don't want to re-fit on test data or live traffic.

Label mapping is handled separately in dataset_config.py. Different datasets call the same attack type different things, so there's a unified 5-class scheme (UNIFIED_5CLASS) for cross-dataset comparisons, but the model's family head outputs 7 classes. You need to be explicit about which mapping you're using when computing metrics.

The Model

Architecture — HelixIDSFull

The model (src/helix_ids/models/helix_ids_full.py) is a multi-task network. One input, two output heads.

Figure 1 — The model takes a 17-dim feature vector, runs it through a shared backbone, then splits into two heads. The family head's 7-class logits go to the HELIX runtime. The binary head is secondary (used for monitoring, not override decisions).

Logits and Softmax

After the family head, you get a vector of 7 raw numbers — one per attack class. These are called logits. They're unconstrained; they can be negative or in the tens of thousands. Higher = stronger prediction for that class.

To turn those into probabilities, you apply softmax:

$$p_i = \frac{\exp(z_i)}{\sum_j \exp(z_j)}$$

Each probability pᵢ is the exponentiated logit for class i divided by the sum of all exponentiated logits. The result is a valid probability vector (sums to 1). The predicted class is just argmax(p) — the class with the highest probability.

Before softmax, the runtime subtracts the maximum logit from all logits (max-normalisation). This keeps the exponentiation numerically stable without changing the argmax. An optional temperature parameter (fixed_temperature) can scale the logits to widen or narrow the distribution before softmax.

The HELIX Runtime

The runtime lives in src/helix_ids/operations/inference_runtime.py as HelixInferenceRuntime. Everything it does is configurable through InferenceConfig.

There are two override mechanisms, applied in sequence:

1. Global coverage override (optional)

If the model's top probability is below a coverage threshold for any sample, that sample gets redirected to a fallback class. This is a blunt instrument — it handles the "model isn't sure about anything" case.

2. Class margin override (the main event)

This is where most of the interesting behaviour lives. For a configured target class (default: class 4), the runtime checks whether the model picked that class with enough conviction. If the margin is too small, it switches the prediction to the second-best class.

Decision Margin

The margin is how far ahead the top prediction is from the runner-up. In logit space:

$$m = z_{(1)} - z_{(2)}$$

z₍₁₎ is the highest logit; z₍₂₎ is the second-highest. A large margin means the model is confident — the winning class is way ahead. A small margin means it's a close call. The override fires when this margin falls below a threshold.

In the code: margin = top1_val - top2_val_only, computed from torch.topk(logits, k=2, dim=1).

You can also work in the z-score domain instead of raw margin. When class_margin_override_use_margin_zscore = True, the margin gets normalised using the mean and standard deviation of a running buffer:

$$s(m) = \frac{m - \mu}{\sigma}$$

μ and σ come from the rolling margin buffer. A small epsilon is added to σ to avoid division by zero. Working in z-score space makes thresholds dataset-independent — a threshold of −0.5 means "half a standard deviation below the buffer mean" regardless of whether raw margins are in the hundreds or hundreds of thousands.

Thresholds — Fixed and Adaptive

There are two threshold types, and you can use either or both:

Fixed threshold (τf)

Set via class_margin_override_tau (default: 70,000.0). It never changes. Simple, predictable. The override fires when margin < τf. If you're using z-score mode, the raw tau is converted to the normalised domain at load time so the comparison is always in the same units.

Adaptive threshold (τa)

Enabled by class_margin_override_use_percentile = True. The threshold is the k-th percentile of a rolling buffer of recent margins for samples where the target class was predicted:

$$\tau_a = Q_k\!\left(\{m_i\}\right)$$

Qₖ is the empirical k-th percentile of the buffered margins. k is set by class_margin_override_percentile_k (default: 75). Buffer size: class_margin_override_buffer_size (default: 1000). The buffer only contains margins from samples where the target class was the argmax prediction. Warmup kicks in after class_margin_override_warmup_min_samples (default: 200) — before that, adaptive mode is inactive.

The Override Rule

When hybrid AND mode is active (class_margin_override_hybrid_and = True), both thresholds have to agree before an override fires:

$$\text{override} = \underbrace{\mathbf{1}\{\text{argmax} = c\}}_{\text{target class predicted}} \;\wedge\; \underbrace{\mathbf{1}\{s(m) < \tau_f\}}_{\text{below fixed threshold}} \;\wedge\; \underbrace{\mathbf{1}\{s(m) < \tau_a\}}_{\text{below adaptive threshold}}$$

c is the target class (class_margin_override_class_id, default 4). All three conditions must hold simultaneously. When the override fires, the prediction is replaced with the second-best class. The adaptive threshold clause drops out if warmup hasn't completed yet.

Safety Guards

Two guards prevent the system from going off the rails:

GuardWhat it watchesWhat it does
Override rate guard Rolling override rate in a sliding window deque If rate > class_margin_override_rate_guard_threshold (default 0.5), freezes the adaptive threshold to stop escalation
Precision guard Live precision on labelled traffic vs baseline If precision drops below baseline × class_margin_override_precision_disable_ratio, disables margin overrides entirely
Collapse alert Variance of buffer values If variance ≤ class_margin_buffer_variance_epsilon, sets class_margin_collapse_alert = True for downstream monitoring to act on

The short story: anything that uses a live adaptive threshold without freezing it eventually collapses. The fix — covered in Section 7 — is to freeze the threshold from calibration data.

What Broke and Why

During evaluation, online_hybrid produced zero overrides on both UNSW and NSL test sets. The model still predicted class 4 instances. The HELIX layer just wasn't doing anything — override_rate = 0.0, KL divergence between pre- and post-HELIX distributions = 0.0.

This was named "control collapse." The controller was technically running, but it had stabilised at a useless equilibrium.

The Feedback Loop Problem

Here's what goes wrong. The adaptive threshold τa is the k-th percentile of the margin buffer. The margin buffer is populated by margins from samples where the model predicted class 4. But whether a sample's prediction gets overridden affects what ends up in the buffer on future batches. The threshold is computed from the same stream it's supposed to gate.

Figure 2 — The closed feedback loop. The adaptive threshold τa is the percentile of the margin buffer. The buffer is fed by the same prediction stream the threshold is supposed to gate. The threshold chases the distribution, eliminating violations. The controller goes quiet.

Think of it this way: by definition, roughly k% of the buffer values fall below the k-th percentile. If the AND condition also requires margins to fall below the fixed threshold τf, and the distribution drifts so that τa tracks the region near τf, the intersection of the two conditions converges to zero. No overrides fire, the buffer stays representative of the full class 4 distribution, τa keeps adapting to match that distribution, and nothing ever changes. Stable, inactive, useless.

⚠️
Control collapse signs: override_rate = 0.0, KL divergence = 0.0, model still predicting class 4 instances but HELIX producing no effect. The bootstrap CI for the collapsed variants was exactly [0.0, 0.0] — not even noise, structurally zero.

The Fix

The fix breaks the feedback loop by making τa independent of the live data stream. Instead of computing the adaptive threshold online, you freeze it to a value computed once on a held-out calibration split.

Two things together

1. Freeze the adaptive threshold. Set class_margin_override_freeze_adaptive_tau = True and provide a calibrated constant via class_margin_override_frozen_tau_adaptive. This constant is computed on calibration data before deployment, not updated at runtime.

2. Use z-score normalisation. Set class_margin_override_use_margin_zscore = True. This puts both the frozen threshold and the runtime margins on the same normalised scale, making the threshold valid across different operating conditions.

At load time, the frozen threshold is converted once to the z-score domain:

$$s\!\left(\tau_a^{\text{frozen}}\right) = \frac{\tau_a^{\text{frozen}} - \mu}{\sigma}$$

μ and σ are the calibration split statistics. This scalar is computed once. At runtime, each incoming margin m is normalised the same way as s(m) = (m − μ)/σ and compared to this fixed value. Both sides of the comparison share the same normalisation so the units always match.

The calibration values from the UNSW experiment (in finding-structured.json):

mu                   = 67137.24
sigma                = 19616.67
tau_adaptive_frozen  = 81921.39

Substituting: $s(\tau_a^{\text{frozen}}) = (81921.39 - 67137.24)\,/\,19616.67 \approx 0.754$. So the override fires when the incoming margin z-score is below ~0.754 — roughly the 75th percentile of the calibration distribution, which is exactly what you'd expect for a percentile_k of 75.

Why this works

With τa frozen, the threshold is a constant with respect to the live stream. No feedback. The override decision for a given sample depends only on that sample's margin, the calibration stats (μ, σ), and the fixed threshold value. The system's past predictions can't shift the goalposts.

Figure 3 — With the frozen fix. The adaptive threshold is loaded once from calibration data and never changes. The threshold gate does a straight comparison between the z-score margin and two fixed values. No part of the decision feeds back to influence future thresholds.

How to deploy this

Pass these flags when exporting the inference bundle via scripts/operations/export_inference_bundle.py:

--class-margin-override-frozen-tau-adaptive 81921.39
--class-margin-override-use-margin-zscore
--class-margin-override-freeze-adaptive-tau

The same flags are available in scripts/operations/serve_rest.py for live serving.

A few things worth noting:

  • frozen_z_hybrid matches fixed_tau_only exactly on UNSW — same override rate to six decimal places, same macro_F1. The frozen calibration successfully approximates the fixed threshold in z-score units.
  • The collapse is total, not partial. A CI of [0.0, 0.0] across 1,000 resamples means zero variance — the controller is deterministically inactive. No subset of the test data could trigger an override.
  • NSL override rate blows up (0.378) for both active variants. The UNSW-calibrated threshold doesn't transfer well to NSL's class 4 margin distribution. The deployment gate (2%) would fail hard here — see the next section.

Deployment Notes

The runbook is in docs/OPERATIONS_DEPLOYMENT_RUNBOOK.md. The short version of what matters:

The 2% override rate gate

The REST server (scripts/operations/serve_rest.py) exports a Prometheus metric helix_degraded_state, which is 1 when override_rate exceeds 2%. If that flag is up, staged rollout stops. The check is in _degraded().

This creates an immediate problem: the NSL test results showed override_rate = 0.378. That's 19x the gate. So you can't use UNSW calibration stats for NSL-like traffic — you need to re-run the calibration on data representative of your actual deployment environment.

Metrics to watch

Live events

Every prediction is logged to artifacts/operations/live_events.jsonl by the REST server. This file is what the staging gate checks and what you'd feed to any post-deployment analysis.

Baselines

Before staging, freeze operational baselines via src/helix_ids/operations/baseline_freeze.py. These are the reference values the precision guard compares against. If live precision degrades below a fraction of the baseline, overrides shut off.

⚠️
Precision guard needs labels. This guard only works if a subset of live traffic has ground-truth labels. In fully unlabelled deployments, the override rate guard is your only active safety mechanism.

Known Limitations

  • 🔁 Calibration doesn't transfer cross-dataset. A frozen threshold calibrated on UNSW produces a 37.8% override rate on NSL. You need dataset-specific calibration for each deployment environment.
  • 🎯 One target class at a time. The current implementation targets a single class (class_margin_override_class_id). There's no built-in support for parallel per-class policies.
  • 🔢 input_dim mismatch. helix_full_config.py says 18; helix_ids_full.py says 17. Reconcile these before doing anything that touches feature dimension calculations.
  • 🏷️ Label scheme mismatch. Family head outputs 7 classes; UNIFIED_5CLASS uses 5. Make sure your evaluation code uses a consistent mapping or you'll get silently wrong per-class metrics.
  • 📊 Benchmark datasets aren't real traffic. NSL-KDD and UNSW-NB15 are lab-generated. Margin distributions in production will probably look different. Budget for a recalibration pass once you have real deployment data.
  • 🔬 Bootstrap validation is offline-only. The 1,000-resample bootstrap in finding-structured.json runs on full test sets. There's no rolling CI in the production monitoring stack yet.
